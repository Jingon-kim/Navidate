# NaviDate 시장조사 보고서

> 조사일: 2025년 1월
> 목적: 커플 전용 개인화 데이트 추천 서비스 개발을 위한 시장 분석

---

## 1. 경쟁 서비스 분석

### 1.1 커플 전용 앱 (기록/소통 중심)

| 서비스 | 핵심 기능 | 강점 | 약점 |
|--------|----------|------|------|
| **비트윈 (Between)** | 커플 채팅, 사진 저장, 기념일 관리, 위치 공유 | 국내 1위, 클라우드 저장, D-day 위젯 | 데이트 추천 기능 없음, 성장 정체 |
| **썸원 (Someone)** | 커플 다이어리, 기념일, 스티커 | 10~20대 초반 인기 | 개인화 추천 부재 |
| **기억나무** | 커플 다이어리, 게임, 운세 | 감성적 UI | 실용적 기능 부족 |

**인사이트**: 기존 커플 앱들은 "기록"에 집중. **"추천"과 "학습"은 블루오션**.

---

### 1.2 데이트 플래닝 앱 (추천 중심)

| 서비스 | 핵심 기능 | 개인화 수준 | 특징 |
|--------|----------|-------------|------|
| **Cupla** | AI 데이트 제안, 캘린더 동기화, 공동 플래닝 | 중간 | 일정 기반 추천, 목표 설정 |
| **Cobble** | 스와이프 매칭, 주 150개 아이디어 추가 | 낮음 | 둘이 스와이프해서 매칭 |
| **Lovewick** | 데이트 아이디어, 협업 플래닝 | 중간 | "로맨스 컨시어지" 컨셉 |
| **LoveTrack** | 500+ 데이트 아이디어, 단계별 가이드 | 낮음 | 콘텐츠 풍부, 개인화 약함 |
| **CoupleQuest** | 맞춤 데이트 플랜 생성 | 중간 | 입력 기반 큐레이션 |
| **DUO** | 스와이프 기반, 서프라이즈 데이트 | 중간 | 전문 플래너 연결 |

**인사이트**: 대부분 **일회성 입력 기반**. 지속적 피드백 학습 시스템은 **전무**.

---

### 1.3 개인화 추천 시스템 기술 동향

#### 추천 알고리즘 유형

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **협업 필터링** | 비슷한 사용자/아이템 기반 추천 | 정확도 높음 | 콜드스타트 문제 |
| **콘텐츠 기반 필터링** | 아이템 특성 기반 추천 | 신규 아이템 즉시 추천 가능 | 다양성 부족 |
| **하이브리드** | 위 두 가지 결합 | 균형 잡힌 추천 | 복잡도 증가 |
| **강화학습 기반** | 피드백으로 지속 학습 | 장기 최적화 | 초기 데이터 필요 |

#### 피드백 유형

- **명시적 피드백**: 평점, 좋아요, 리뷰
- **암시적 피드백**: 클릭, 체류시간, 재방문

**인사이트**: 커플 2명의 **암시적+명시적 피드백** 결합이 핵심 차별화 포인트.

## 2. 시장 기회 분석

### 2.1 기존 서비스의 공통적 한계

```
❌ 일회성 추천 (피드백 미반영)
❌ 개인보다 "일반 커플" 대상
❌ 날씨, 예산, 시간 등 맥락 미반영
❌ 두 사람의 취향 교집합 미분석
❌ 데이트 후 피드백 → 학습 루프 없음
```

### 2.2 NaviDate 차별화 기회

```
✅ 우리 둘만을 위한 1:1 개인화
✅ 데이트 후 피드백 → AI 학습 → 점점 정확해지는 추천
✅ 날씨/예산/시간/컨디션 등 맥락 반영
✅ 두 사람 취향의 교집합 & 새로운 시도 밸런스
✅ 히스토리 축적 → "우리만의 데이트 아카이브"
```

## 3. 타겟 사용자 정의

### 3.1 Primary Persona: "우리 커플"

```yaml
사용자: 본인 + 파트너 (2명 한정)
관계 기간: 다양 (초기 연애 ~ 장기 커플/부부)
핵심 니즈:
  - "이번 주말 뭐하지?" 해결
  - 매번 비슷한 데이트 탈피
  - 서로의 취향 발견 & 존중
  - 추억 기록 + 미래 계획
```

### 3.2 사용 시나리오

| 시나리오 | 입력 | 기대 출력 |
|----------|------|-----------|
| 주말 데이트 | "토요일 오후, 홍대 근처, 10만원 이하" | 맞춤 코스 3개 |
| 기념일 | "100일 기념, 특별하게" | 스페셜 코스 + 예약 링크 |
| 급 계획 | "지금 당장, 1시간 안에" | 현 위치 기반 즉석 추천 |
| 새로운 시도 | "안 가본 스타일로" | 취향 확장 추천 |

## 4. 핵심 기능 요구사항

### 4.1 Must Have (MVP)

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| **커플 프로필** | 각자 취향, 싫어하는 것, 제약사항 입력 | P0 |
| **데이트 추천** | 조건 입력 → AI 코스 생성 | P0 |
| **피드백 수집** | 데이트 후 평가 (별점 + 코멘트) | P0 |
| **학습 반영** | 피드백 기반 추천 정확도 향상 | P0 |
| **히스토리** | 과거 데이트 기록 열람 | P1 |

### 4.2 Nice to Have (확장)

| 기능 | 설명 |
|------|------|
| 캘린더 연동 | 일정 기반 자동 추천 |
| 예산 트래킹 | 월별 데이트 비용 관리 |
| 서프라이즈 모드 | 한 명이 몰래 계획 |
| 기념일 알림 | D-day 기반 특별 추천 |

## 5. 기술 아키텍처 권장사항

### 5.1 데이터 모델

```
[커플]
├── 파트너 A 프로필
│   ├── 선호 카테고리 (음식, 활동, 분위기)
│   ├── 싫어하는 것 (알러지, 공포증 등)
│   └── 컨디션 패턴 (주중/주말, 계절별)
├── 파트너 B 프로필
│   └── (동일 구조)
├── 공통 선호
│   ├── 예산 범위
│   ├── 이동 수단
│   └── 선호 지역
└── 데이트 히스토리
    ├── 날짜, 장소, 코스
    ├── 각자 평점 + 코멘트
    └── 사진 (선택)
```

### 5.2 추천 엔진 설계

```
[입력]
날짜/시간 + 지역 + 예산 + 분위기 + 현재 컨디션
            ↓
[필터링 레이어]
날씨 체크 → 영업시간 체크 → 예산 필터 → 거리 필터
            ↓
[개인화 레이어]
A 선호 점수 × B 선호 점수 × 히스토리 가중치
            ↓
[다양성 레이어]
최근 방문 페널티 + 새로운 시도 보너스
            ↓
[출력]
코스 옵션 3개 (안전한 선택 / 균형 / 모험적)
```

## 6. 성공 지표 (KPI)

| 지표 | 설명 | 목표 |
|------|------|------|
| **추천 채택률** | 추천 코스 중 실제 실행 비율 | 60%+ |
| **피드백 제출률** | 데이트 후 평가 작성 비율 | 70%+ |
| **평균 만족도** | 데이트 후 평점 평균 | 4.0/5.0+ |
| **재사용률** | 월간 서비스 재사용 | 주 1회+ |
| **추천 정확도 향상** | 피드백 반영 후 만족도 증가 | 월 5%+ |

## 8. 결론 및 권장사항

### 핵심 차별화 전략

> **"학습하는 커플 전용 AI"**
> - 대중이 아닌 "우리 둘"만을 위한 서비스
> - 쓸수록 똑똑해지는 추천 엔진
> - 데이트 히스토리가 곧 우리의 자산

### 즉시 실행 가능한 다음 단계

1. ✅ 시장조사 완료 (본 문서)
2. ⏳ PM 에이전트 구축 (판단 프레임워크)
3. ⏳ MVP 기능 명세서 작성
4. ⏳ 프로토타입 개발
