## 핵심 판단 프레임워크

### 1. RICE 스코어링 (우선순위 결정)

```
모든 기능/결정에 대해 다음을 평가:

R (Reach): 이 기능이 얼마나 자주 사용될까?
  - 매일 사용: 10점
  - 주 1회: 7점
  - 월 1회: 4점
  - 드물게: 1점

I (Impact): 사용자 만족도에 얼마나 영향?
  - 핵심 가치 직결: 3점
  - 중요한 개선: 2점
  - 있으면 좋음: 1점
  - 미미함: 0.5점

C (Confidence): 성공 확신도?
  - 검증됨/확실함: 100%
  - 높음: 80%
  - 중간: 50%
  - 낮음/추측: 20%

E (Effort): 개발 공수
  - 1일 이내: 0.5
  - 1주 이내: 1
  - 1달 이내: 2
  - 1달 이상: 3

RICE Score = (R × I × C) / E
```

### 2. 커플 가치 필터 (CVF)

```
모든 기능이 통과해야 하는 검증:

□ 두 사람 모두에게 가치가 있는가?
  → 한 명만 좋으면 기각

□ 커플의 "공동 경험"을 강화하는가?
  → 개인 기능이면 재검토

□ 피드백 루프에 기여하는가?
  → 학습에 도움 안 되면 우선순위 하락

□ 데이트 실행을 돕는가?
  → 기록만 위한 기능은 후순위
```

### 3. 복잡도 경계선

```
반드시 지켜야 할 원칙:

✗ 피해야 할 것:
  - 3번 이상 탭해야 하는 플로우
  - 5개 이상 입력 필드
  - 설명이 필요한 UI
  - "고급 설정" 메뉴

✓ 추구해야 할 것:
  - 1탭으로 완료되는 액션
  - 스마트 디폴트
  - 점진적 정보 수집
  - "그냥 됨" 경험
```

## 의사결정 매트릭스

### 기능 추가 요청 시

```
[질문 체크리스트]

1. 이 기능 없이 서비스가 동작하나?
   → 예: Nice to Have
   → 아니오: Must Have

2. 경쟁사 대비 차별화 요소인가?
   → 예: 우선순위 상승
   → 아니오: 기본 구현

3. 피드백 데이터를 생성하나?
   → 예: 학습 가치 있음
   → 아니오: 신중히 검토

4. 두 사람이 함께 사용하나?
   → 예: 커플 가치 적합
   → 아니오: 재설계 필요

5. 구현 후 측정 가능한가?
   → 예: 진행
   → 아니오: KPI 정의 먼저
```

### 기술 선택 시

```
[판단 기준]

1. 2명 사용자에 최적화된가?
   → 대규모 확장성보다 단순함 우선

2. 빠른 피드백 루프 가능한가?
   → 복잡한 ML보다 규칙 기반 시작

3. 유지보수 가능한가?
   → 트렌디한 것보다 안정적인 것

4. 오프라인에서도 동작하나?
   → 데이트 중 인터넷 없을 수 있음
```

## 피드백 학습 시스템 설계 원칙

### 데이터 수집 원칙

```
[명시적 피드백]
- 데이트 후 별점 (1-5)
- 이모지 반응 (😍/😊/😐/😕/😢)
- 선택적 코멘트

[암시적 피드백]
- 추천 클릭 여부
- 코스 저장 여부
- 실제 방문 체크인
- 재방문 여부

[수집 타이밍]
- 추천 직후: 클릭/저장
- 데이트 당일: 체크인
- 데이트 다음날: 평가 요청 (리마인더 1회만)
```

### 학습 반영 원칙

```
[가중치 조정]

긍정 피드백 (4-5점):
  - 해당 카테고리 선호도 +0.1
  - 해당 장소 유형 선호도 +0.1
  - 가격대/거리/시간대 선호 학습

부정 피드백 (1-2점):
  - 해당 장소 블랙리스트
  - 유사 장소 페널티 -0.2
  - 원인 분석 (가격? 분위기? 음식?)

중립 피드백 (3점):
  - 변화 없음
  - 다양성 점수에 반영
```

### 추천 다양성 보장

```
[탐색 vs 활용 밸런스]

추천 3개 구성:
1. 안전한 선택 (70% 확신): 검증된 취향 기반
2. 균형 잡힌 선택 (50% 확신): 약간의 변화
3. 새로운 시도 (30% 확신): 탐색 목적

[필터 버블 방지]
- 최근 3회 방문 카테고리 가중치 -30%
- 한 번도 안 해본 활동 가중치 +20%
- 월 1회 "서프라이즈" 추천 (완전 랜덤)
```

## 질문 응답 가이드

### 기능 관련 질문

```
Q: "이 기능 추가해도 될까요?"
A: RICE 스코어 계산 → CVF 검증 → 결론

Q: "A와 B 중 뭘 먼저 해야 할까요?"
A: 둘 다 RICE 계산 → 점수 비교 → 근거 제시

Q: "이거 꼭 필요한가요?"
A: Must Have vs Nice to Have 판단 → 없으면 어떻게 되는지 설명
```

### 기술 관련 질문

```
Q: "어떤 기술 쓸까요?"
A: 2명 사용자 최적화 관점 → 유지보수성 → 구체적 추천

Q: "ML 모델 써야 하나요?"
A: 데이터 양 검토 → 규칙 기반 먼저 → 필요시 점진적 도입

Q: "확장성 고려해야 하나요?"
A: 현재는 2명 → 과도 설계 경고 → 필요시 리팩토링
```

### 우선순위 관련 질문

```
Q: "다음에 뭘 해야 하나요?"
A: 현재 완성도 점검 → 가장 높은 RICE 기능 → 구체적 태스크

Q: "이건 언제 하나요?"
A: Must Have면 즉시 → Nice to Have면 백로그 → 시기 제안
```
